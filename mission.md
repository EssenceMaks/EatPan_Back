# Архитектурное решение: High Availability (HA) инфраструктура для EatPan

## 📌 Контекст и Цели
Проект EatPan требует создания отказоустойчивой, масштабируемой (Enterprise-level) архитектуры с минимальными затратами на старте. 
Основная вычислительная мощность базируется на локальном мощном железе разработчика (Node 1: i7, 24GB RAM). Облачные решения используются исключительно как резерв (Failover) на случай проблем с питанием или локальным провайдером интернета.

## 🛠 Выбранный стек технологий
* **Балансировка / DNS:** Amazon Route 53, Cloudflare (Primary), Hetzner VPS с Nginx (Secondary).
* **Бэкенд:** Django API (Stateless, упакован в Docker).
* **Кэширование:** Valkey (Форк Redis, открытая лицензия BSD 3-Clause, 100% совместимость с `django-redis`). Снижает нагрузку на SSD и БД.
* **База данных:** Supabase (Self-hosted PostgreSQL). Безлимитное коммерческое использование.

## 🏗 Эволюция решений и Итоговая Архитектура

### Уровень 1: Умный DNS (Маршрутизация)
**Проблема:** Как перенаправить пользователей, если упадет основной балансировщик (Cloudflare)?
**Решение:** Используем Amazon Route 53 (Health Checks). 
* Route 53 шлет 100% трафика на Cloudflare. 
* Если Cloudflare лежит, Route 53 автоматически переключает DNS на резервный дешевый VPS (Hetzner) с Nginx.

### Уровень 2: Балансировщики (Точки входа)
**Проблема:** Нельзя светить домашний IP-адрес напрямую в интернет из-за безопасности и проблем с динамическим IP.
**Решение:** Cloudflare Tunnels (Primary) и Tailscale/WireGuard (Secondary). Балансировщики принимают внешний трафик и по защищенным туннелям пробрасывают его на вычислительные узлы. Они работают по принципу Active-Passive: 100% запросов идут на домашний ПК, пока он в сети.

### Уровень 3: Вычислительные узлы (Бэкенд + Кэш)
**Проблема:** Оптимизация затрат и скорости. 
**Решение:** * **Node 1 (Домашний ПК - Active):** Берет на себя всю нагрузку. Здесь в отдельных Docker-контейнерах крутятся Django API и Valkey (для In-Memory кэширования сессий, rate-лимитов и частых запросов).
* **Node 2 (Облако, например Render - Passive):** Спящий резерв. Просыпается и принимает трафик от балансировщика, только если Node 1 отключается от сети.

### Уровень 4: База данных (Supabase) и предотвращение Split-Brain
**Проблема:** Если сделать репликацию Master-Master или Master-Replica без арбитража, при обрыве интернета домашнего ПК система рухнет или создаст две конфликтующие базы (Split-Brain). База не может просто так "передавать флаг".
**Решение (Leader Election & Failback):**
* Внедряется "Судья" (Consul / Arbiter / Patroni) на независимом дешевом облачном сервере.
* **База 1 (Домашний ПК):** Изначально владеет "Флагом Мастера" (Primary DB). Принимает чтение и запись.
* **База 2 (Облачный резерв):** Работает как Replica (только чтение).
* **Сценарий отказа:** Если дома пропадает интернет, Судья забирает флаг у Базы 1 и отдает Базе 2. База 2 становится Мастером, резервный Django (Node 2) пишет в нее.
* **Сценарий восстановления (Failback):** Когда домашний ПК возвращается в сеть, он связывается с Судьей, понимает, что он теперь Реплика, скачивает пропущенные данные (через `pg_rewind`), синхронизируется, после чего Судья возвращает ему "Флаг Мастера".

## 💡 Резюме для ИИ-агентов
Все сервисы строго изолированы (один процесс — один контейнер). Инфраструктура разворачивается через `docker-compose` с использованием профилей (Profiles) для удобного запуска специфичных ролей на разных машинах. Код бэкенда не должен содержать состояния (Stateless), все стейты хранятся в Valkey и Supabase.
